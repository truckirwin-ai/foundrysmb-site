# Client Measurement Pack

Use this pack to prove whether a workflow is paying.

## Measurement rule

Do not measure AI activity. Measure business movement.

Weak measures:

- prompts used
- documents generated
- tools adopted
- automations created

Better measures:

- hours recovered
- response time reduced
- bids sent faster
- follow-ups completed
- reports delivered on time
- fewer rework loops
- less owner review time
- more consistent customer experience

## Measurement ladder

Use three layers.

| Layer | Question | Examples |
| --- | --- | --- |
| Activity | Did the workflow run? | runs, users, outputs reviewed |
| Operating movement | Did the work improve? | cycle time, rework, completeness, owner time |
| Business movement | Did the business benefit? | faster bids, recovered leads, better retention, more capacity |

Activity is useful for adoption. Operating movement proves the workflow is working. Business movement supports renewal, install, and managed ops.

## Baseline interview

Ask before the sprint or install:

- What makes this workflow painful?
- What happens when it is late?
- What happens when it is wrong?
- Who gets interrupted?
- How often does the owner step in?
- What does the customer experience?
- What work gets delayed because of this?
- What would "better" look like in 30 days?
- What would make this not worth continuing?

## KPI menu

| Workflow | Primary KPI | Secondary KPI |
| --- | --- | --- |
| Bid prep | Time from lead to bid-ready packet | Missing-scope questions caught |
| Lead follow-up | Time to first useful response | Follow-ups completed on schedule |
| Reporting | Time to draft report | Reports delivered on time |
| SOP capture | SOPs completed and used | New-hire/operator questions reduced |
| Context vault | Good outputs from approved context | Missing-context issues found |
| Inbox triage | Owner inbox time | Tasks routed correctly |

## KPI quality test

A useful KPI is:

- tied to the workflow
- visible within 30 days
- understandable by the owner
- hard to fake with activity alone
- connected to a decision
- cheap enough to collect

Avoid KPIs that require a new reporting system before the workflow has proven value.

## Scorecard interpretation

| Pattern | Interpretation | Action |
| --- | --- | --- |
| Usage up, value unclear | People are trying it, but metric is weak | Improve measurement or workflow target |
| Cycle time down, rework up | Faster but lower quality | Tighten review gate |
| Owner time down, team confusion up | Delegation improved but SOP weak | Improve handoff/training |
| Quality up, usage low | Good asset, adoption issue | Observe users and remove friction |
| Value up, context gaps recurring | Workflow is worth managed ops | Propose recurring context maintenance |
| No movement after 30 days | Wrong workflow or poor adoption | Stop, redesign, or choose new workflow |

## Owner-time recovery model

Owner time matters more than generic labor time.

Track:

- owner interruptions per week
- minutes per interruption
- decisions owner no longer has to reconstruct
- approvals made easier by better packets
- customer responses owner no longer rewrites
- reports owner no longer assembles

Owner-time proof statement:

```text
Before, the owner spent [X] hours/week reconstructing [workflow].
After, the owner reviews [output] with [context/review packet] and spends [Y] hours/week.
Recovered owner capacity: [X-Y] hours/week.
```

## Renewal proof checklist

Managed Ops should renew only if there is proof of value.

- [ ] The workflow ran repeatedly.
- [ ] The owner can name what improved.
- [ ] Metrics show time, quality, speed, or consistency improvement.
- [ ] The context vault improved.
- [ ] The SOP/prompt chain improved.
- [ ] The next workflow backlog is clearer.
- [ ] The client knows what next month is for.

## Renewal narrative

Use this structure before asking for renewal or expansion:

```text
This month, the workflow ran [volume].

The main improvement was [time/speed/quality/owner load/revenue support].

We changed [context/prompt/SOP/review/measurement].

The strongest evidence is [metric or example].

The remaining bottleneck is [next issue].

Next month should focus on [managed ops focus] because [business reason].
```

## Decision model

| Evidence | Decision |
| --- | --- |
| Clear improvement and active use | Continue managed ops |
| Improvement but low adoption | Focus next month on adoption |
| High value but workflow still rough | Run focused improvement sprint |
| No measurable value | Stop or redesign the workflow |
| New workflow now more valuable | Move support focus |

## Client-facing proof packet

For a paid client, assemble:

- baseline snapshot
- current scorecard
- before/after example
- change log
- business impact summary
- next-month recommendation
- backlog
- renewal decision

The proof packet is not decoration. It is the reason the buyer keeps funding the work.
